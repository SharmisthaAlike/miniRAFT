Mini-RAFT project execution plan across three weeks, broken into phases with daily tasks per person and dependency tracking.

Day 1–2 All 4 together — agree on contracts before anyone writes code

This is the most important session of the entire project. Every interface between your components must be pinned down here so nobody has to wait on anyone else later. Spend a full session (2–3 hrs) on this together.

Stroke message format

Agree on the JSON shape a stroke takes. Example:
{"id","x0","y0","x1","y1","color","clientId"}
P1, P2, P4 all depend on this.

RPC endpoint shapes

Pin POST /request-vote, POST /append-entries, POST /heartbeat, POST /sync-log — request + response JSON for each. P1 implements, P2 + P3 call them.

Leader advertisement

How does Gateway know who the leader is? Agree on one mechanism: replicas expose GET /status returning {"leader","term","id"}. P1 implements, P2 polls it.

Gateway → client WebSocket events

Agree on event names and shapes. Example: stroke_committed, leader_changed. P2 emits, P4 consumes.

Replica IDs + ports

Fix replica IDs (r1, r2, r3), ports (3001, 3002, 3003), gateway port (3000). P3 sets these in docker-compose, everyone else hardcodes nothing — reads from env.

Log entry schema

What goes in a log entry: {"index","term","stroke"}. P1 owns this, but P3's sync-log and P2's broadcast both depend on it.

Exit gate: One shared doc (Google Doc or repo README) with all 6 contracts written and signed off by all 4. Nobody writes production code until this exists.

Day 8–10 Core logic — parallel, no blocking
P1 — RAFT core
Election timerFollower starts random 500–800ms timer. On expiry → become Candidate, increment term, request votes from peers.
RequestVote handlerGrant vote if term is higher and haven't voted this term. Respond with {term, voteGranted}.
Become leaderOn majority votes → set role to Leader. Start heartbeat loop at 150ms.
AppendEntries (leader side)On incoming stroke: append to log, fan out AppendEntries to both followers, wait for acks, commit when majority respond.
P2 — Gateway
Leader poller livePoll /status on replicas every 200ms. Track current leader. Re-poll immediately on any non-200 from leader.
Stroke forwardingOn WebSocket message from browser: POST stroke to leader's /append-entries.
Committed stroke broadcastLeader calls gateway's POST /broadcast (or gateway polls — agree which). Gateway fans out to all connected WebSocket clients.
P3 — Infra
AppendEntries (follower side)Receive AppendEntries: validate term + prevLogIndex. Append entry to log. Respond with {success, logLength}.
Heartbeat handlerReset election timer on valid heartbeat. Drop if term is stale.
/sync-log endpointAccept {fromIndex}. Return all committed entries from that index onward.
Catch-up on restartOn first AppendEntries with prevLogIndex mismatch: respond with current log length. Receive sync-log response, append all missing entries.
P4 — Frontend
Draw + sendCanvas strokes POST to gateway via WebSocket. No double-draw — only render what comes back from committed broadcast.
Receive + paintOn stroke_committed WebSocket event: draw the stroke on canvas. Works for both own strokes and remote ones.
Multi-tab testOpen 2 tabs manually. Draw in one. Verify the stroke appears in both via the gateway broadcast path.
Watch out: P2 needs P1's /status endpoint to return real leader info. If P1 isn't ready, P2 can hardcode r1 as leader temporarily — but agree on this explicitly and revert it as soon as P1's election works.
Day 11–14 Integration — connect the pipes and verify end-to-end
P1
Election under real loadKill a replica manually. Verify remaining two elect a new leader within 1–2 seconds. Watch logs for term increments.
Stale leader detectionIf a new leader is elected, old leader receiving a heartbeat/vote with higher term must step down to Follower immediately.
P2
Failover routingKill the current leader. Verify gateway detects new leader (via /status polling) and routes subsequent strokes there. No client disconnect.
Buffering during electionIf no leader is found for <2s, buffer incoming strokes and replay them once a leader is confirmed.
P3
Restart + catch-up testStop replica 3. Draw 5 strokes in the system. Restart replica 3. Verify it calls /sync-log and recovers all 5 entries. Check log matches leader.
Hot-reload testEdit a file in /replica2. Verify container restarts, rejoins cluster, catches up. No client disconnection.
P4
3-client consistency testOpen 3 tabs. Draw rapidly from all three. After 30s, all canvases must show identical strokes in identical order. Screenshot and diff.
Failover UX testKill leader mid-draw. Verify canvas doesn't freeze or corrupt. Strokes drawn during election should either commit or be visibly pending, never silently dropped.
Exit gate (end of Week 2): Full end-to-end works: browser draw → gateway → leader → replicas → committed → broadcast → all canvases updated. Leader election works when you kill a container. Restarted replica catches up.
Day 15–17 Harden — edge cases, reliability, zero-downtime
P1
Split vote handlingSimulate two candidates in same term. Verify both restart election with incremented term. No infinite loop.
Higher-term ruleAny RPC arriving with term > currentTerm must cause node to immediately revert to Follower and update term.
Safety: no overwriteVerify committed entries are never overwritten even if a new leader is elected. Test by restarting the leader after it commits.
P2
Rapid failoverKill and restart the leader 5 times in 30 seconds. Gateway must never drop a client WebSocket. Strokes must not be lost (or are clearly surfaced as pending).
ObservabilityGateway logs every leader change event with timestamp and new leader ID.
P3
Blue-green swapReplace replica1 entirely (docker-compose stop + rm + up). New container joins cluster, catches up via sync-log, participates in elections. No downtime.
ObservabilityEvery replica logs: current role, term number, and commit index once per second. Helps diagnose issues during demo.
P4
Chaos testScript: kill random replica every 10s for 2 minutes while 3 clients draw continuously. At the end, verify all canvases match. Document results.
Stress test100 strokes/sec from a single client for 30s. Check: no memory leak, no crash, commit latency stays reasonable.
Day 18–19 Demo prep — script the video, rehearse the viva

The video must be 8–10 minutes. Script it as 6 scenes, each showing one thing clearly.

Scene 1 — Normal drawing (P4 records)

3 browser tabs open. Draw from all three. Show strokes appearing on all canvases in real time.

Scene 2 — Kill the leader (P1 explains)

Show replica logs → identify leader → kill it → show election in logs → new leader elected → drawing continues.

Scene 3 — Automatic failover (P2 explains)

Gateway log shows leader change event. Strokes drawn during election either committed or replayed. No client disconnect.

Scene 4 — Hot reload (P3 records)

Edit a file in replica2 → container restarts in logs → sync-log fires → replica2 back in cluster → drawing still works.

Scene 5 — Consistent state after restart (P3+P4)

Stop replica3, draw 10 strokes, restart it. Show sync-log firing. Show replica3's log now matches leader. Compare canvas.

Scene 6 — Chaos (P4 explains)

Run the chaos script. Show system surviving. All 3 canvases identical at the end.
